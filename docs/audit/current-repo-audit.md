## 4. Current generated CSS files

- `app/primitives.css` (695 lines) — `:root` palette vars only. Regenerated by `tokens:data`; **committed**, byte-checked by CI.
- `app/brand.css` (97 lines) — `@theme` axis re-points + `:root` / `.dark` semantic vars aliasing primitives (e.g. `--foreground: var(--tw-neutral-950)`). Regenerated by `brand:build`; committed, CI-checked.
- `app/globals.css` — hand-owned structural layer: radius **calc scale** (`--radius-md: calc(var(--radius) * 0.8)`), font stacks, 5 shadow tokens, and the `@theme inline` `--color-* → var(--*)` mapping that makes `bg-primary` etc. work.
- `app/shadcn.css` — hand-owned keyframes + `@custom-variant` (data-open/closed/checked…).

Resolution chain: `bg-primary` → `--color-primary` (globals) → `--primary` (brand.css) → `--tw-neutral-900` (primitives.css). **`--rdx-*` vars are generated but have zero consumers** (grep-verified in `app/` + `components/`).

## 13. Current hardcoded style risks inside components

Measured on `components/ui/` (grep):

- **Raw palette classes: near-zero.** Single hit: `bg-white` on the slider thumb (`components/ui/slider.tsx:52`) — the only semantic-token violation of the "no bg-white/text-black/border-gray" rule. No raw hex/oklch values in any component.
- **Hardcoded control sizes everywhere (expected shadcn baseline):** `h-8` ×16, `h-9` ×4, `h-10` ×4 + matching `px-*`/`text-sm`/`rounded-md` in cva variants (button.tsx, input.tsx, select.tsx…). This is the Phase 4 component-token surface, not a bug today.
- **122 arbitrary-value utilities across 18 files** — mostly legitimate Radix/vendor plumbing: `w-[var(--radix-*-trigger-width)]` (select/dropdown), recharts selector overrides in chart.tsx, `data-[state]` transforms in sheet/drawer. Low-risk but should be inventoried before Phase 4 tokenization.
- **sidebar.tsx** defines its own CSS-var conventions (`--sidebar-width`, etc.) inline — a de-facto component token precedent to formalize in Phase 4.
- Density is hardcoded (no compact/comfortable modes) — matches v2 gap analysis.

## 14. Compatibility risks if we restructure

Ranked (mitigations in [`docs/v2-migration-audit.md`](../v2-migration-audit.md) §2):

1. **Published flat `exports "./*"`** — folder-per-component (Phase 3) breaks `@npsin-oreo/design-system/button` unless the exports map adds explicit per-component entries (or `./*/index`) in the same release. Requires a version bump + migration notes.
2. **`ds-brand-build` bin + `files` whitelist** — `scripts/build-brand.mjs`, `scripts/lib-oklch.mjs`, root `tokens.json` are runtime API for consumers. They must not move; new pipeline must be parallel, not a replacement, until a major release.
3. **CI token-drift job** (`.github/workflows/ci.yml`) — regenerates brand.css/token-contract.json/primitives.css and fails on any byte diff. Every token-pipeline PR must keep this green or update the job in the same PR.
4. **Brand branches** — merge conflicts on restructure are guaranteed for any branch touching more than brand.config.json; semantic layer must stay derivable from brand.config.json until brands migrate.
5. **Storybook glob + stub aliases** (`.storybook/main.ts`) — component moves and story co-location must update the glob and keep the `@/lib/design-tokens` stub path working; `test-storybook` in CI is the safety net.
6. **shadcn CLI workflow** — `components.json` aliases assume `components/ui/<flat>`; folderization changes where `npx shadcn add` writes and how diffs against upstream are made. Decide the CLI story before Phase 3.
7. **Tailwind namespace collision** — future generated primitive CSS must avoid `--color-*` (taken by the `@theme inline` mapping) and `--tw-*` is Tailwind's internal prefix convention; pick the v2 emission prefix deliberately in Phase 2.
8. **Docs-site imports** (`components/docs/*` → `@/lib/design-tokens` stub, registry.tsx) — restructures can silently break the /docs app even when Storybook passes; `next build` must be part of Phase 3 verification.

## 15. Recommended migration order

Matches the agreed phase plan; each step additive and independently verifiable:

1. **Phase 1 — token structure** *(done)*: `tokens/raw|primitive|semantic|mode|theme` generated from legacy sources; zero consumers; legacy pipeline untouched.
2. **Phase 2 — token build pipeline**: `scripts/tokens/validate.mjs` + `build-css.mjs` (+ contract/TS emit) → `dist/tokens/` (git-ignored at first). **Gate:** diff generated CSS var-for-var against `app/primitives.css` + `app/brand.css` before any consumer swap; choose non-colliding var prefix; add `tokens:validate` to CI without touching the drift job.
3. **Phase 3 — repo structure**: folder-per-component **with** compatibility re-export shims at old paths + explicit per-component `exports` entries; update Storybook glob; verify with `tsc`, `next build`, `test-storybook`; version bump + migration notes.
4. **Phase 4 — component tokens**: start with button/input/card/dialog/badge/select/sheet/table/sidebar; wire cva variants to `--<component>-*` vars; kill the `bg-white` in slider.tsx along the way.
5. **Phase 5 — new components**: only proven P3 gaps (promote date-picker/data-table stories into real components; icon/icon-button land in Phase 6).
6. **Phase 6 — icon system**: `icons/icon-registry.ts` + `Icon`/`IconButton` + codemod the 39 direct lucide imports; keep lucide-react as dependency (non-breaking).
7. **Phase 7 — contracts & gates**: `.designops/*` contracts, forbidden-patterns, audit scripts (`hardcoded-styles`, `component-token-coverage`, `icon-usage`) reporting-first, then enforcing in CI.

Rationale for the order: tokens before structure (values must be provably identical before anything moves), structure before component tokens (so tokens land in final file locations once), icons late (widest blast radius, purely mechanical), contracts last (they codify what the earlier phases produced).
