# tokens/ — canonical token layer (DS v2)

Status: **LIVE.** `app/globals.css` imports `dist/tokens/*.css`, generated
from this hand-authored layer by `npm run tokens:build` (predev/prebuild/storybook
hooks run it automatically). Byte-identity of the published CSS is guarded by
`npm run tokens:css-check` (in CI). The legacy `app/*.css` pipeline was removed in
v2.0.0 — `tokens/**` is the single source.

The ONLY generated file in this tree is `theme/<brand>.json`, produced from the
brand config(s) by `npm run tokens:theme`. After editing `brand.config.json`
(the neutral base) or a `brands/<name>.config.json` overlay, run:

```bash
npm run tokens:theme && npm run tokens:build
```

## Layers

| Dir | Layer | May reference | Consumed by |
|---|---|---|---|
| `raw/` | legacy exports (byte copies) | — | migration only — never components |
| `primitive/` | raw values, no UI meaning (`color.blue.500`, `space.2`) | nothing | semantic/component tokens |
| `semantic/` | UI meaning (`color.primary`, `status.success`) | primitive | components, themes |
| `component/` | per-component decisions (`button.height.md`) — **Phase 4** | semantic + primitive | component variants |
| `mode/` | light/dark/density/contrast overrides | existing token paths | build pipeline |
| `theme/` | brand personality (radius, fonts, accents) | semantic + component paths | brand builds |

## Generated vs hand-owned

Everything here is **hand-authored** except the theme overlays, which
`npm run tokens:theme` regenerates from the brand config(s):

```bash
npm run tokens:theme   # regenerates theme/<brand>.json from brand.config.json + brands/*.config.json
```

| Generated by `tokens:theme` — **never hand-edit** | Hand-owned — edit directly |
|---|---|
| `theme/<brand>.json` (neutral + every `brands/*.config.json`) | `primitive/*.json`, `semantic/*.json`, `component/*.json`, `mode/*.json` |

`raw/*` are frozen historical copies (read by `build-css` for the
`--tw-*`/`--brand-*`/`--rdx-*` compat aliases) — never consumed by components.

Hand-owned files reference generated paths via `{alias}` — the Phase 2 validator
checks that those paths exist. Tokens/modes/themes that nothing consumes yet are
marked `$extensions["looloo.status"] = "proposed"` or `PROPOSED` in the file
`$description`. See `docs/tokens/token-source-strategy.md` for the full policy.

Naming note: DTCG token names must not contain `.`, so fractional spacing steps
are hyphenated — Tailwind `px-2.5` ↔ `{space.2-5}`.

## Format

Practical DTCG: `$type`, `$value`, `$description`, `$deprecated`, `$extensions`.
Aliases use `{dot.path}`. `$extensions` namespaces used:

- `looloo.source` — where the value came from in the legacy export
- `looloo.legacy` — original raw value before normalization (px, hex, `{ref}`)
- `looloo.resolved` — resolved OKLCH for an aliased semantic token
- `looloo.status: "proposed"` — new v2 token not consumed by any component yet

## Phase 1 decisions

1. **Radix palette not promoted.** `rdx-*` has zero consumers in `app/` and
   `components/` — it stays in `raw/legacy.tokens.json` for reference only.
2. **Source prefixes dropped.** `tw-blue-500` → `color.blue.500`;
   brand ramps → `color.brand.primary.500` (the `brand.` segment avoids
   colliding with the *semantic* `color.primary`).
3. **Flat kebab semantic keys.** `primary-foreground` (not
   `primary.foreground`) so token names map 1:1 to the existing CSS custom
   properties and to `token-contract.json`.
4. **px → rem** (÷16) for spacing/radius/font sizes; original value kept in
   `$extensions["looloo.legacy"]`.
5. **`brand.config.json` remains the brand-override input.** `npm run tokens:theme`
   maps it (and each `brands/<name>.config.json`) into `theme/<brand>.json`; run it
   after a brand edit. Semantic tokens stay brand-agnostic (they reference the theme tier).

## Rules (enforced by validation from Phase 2)

- Components never import from `raw/`.
- Semantic tokens must not reference component tokens.
- Component tokens must not reference raw tokens.
- No source names (`tw`, `rdx`, `shadcn`) in canonical token paths.
- Deprecations need a replacement note before deletion.
